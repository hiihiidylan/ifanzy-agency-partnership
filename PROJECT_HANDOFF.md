# iFanzy Agency Partnership 项目交接

## 1. 项目目标与当前版本

本项目先通过 HTML/CSS 完成机构合作演示页面，再由固定 PDF 脚本将 HTML 渲染为可交付 PDF。HTML 是设计与内容的唯一源文件，PDF 是生成产物，不应直接编辑 PDF。

当前重点维护 4:3 版本：

- 英文 HTML：`mobile/agency_partnership_4x3.html`
- 繁中 HTML：`mobile/agency_partnership_4x3_tc.html`
- PDF 脚本：`generate_mobile_pdf.js`
- 英文 PDF：`outputs/pdf/agency_partnership_4x3_en.pdf`
- 繁中 PDF：`outputs/pdf/agency_partnership_4x3_tc.pdf`

仓库根目录中的 4:3 HTML 与工作目录 `mobile/` 中的对应文件保持同步。除非用户明确要求，不要修改以下移动版文件：

- `agency_partnership_mobile.html`
- `agency_partnership_mobile_tc.html`

## 2. 标准工作流

1. 只修改 HTML/CSS 源文件，先完成页面内容、布局和视觉设计。
2. 启动本地服务，在浏览器中逐页预览中英文版本。
3. 根据预览反复调整，确认页面比例、分行、字号、行距和模块边界。
4. 使用 PDF 脚本生成中英文 4:3 PDF。
5. 使用 `pdfinfo` 检查页数和页面尺寸，并把 PDF 页面渲染为 PNG 做视觉检查。
6. 确认无裁切、遮挡、空白页和字体异常后，再同步 HTML、脚本和 PDF 到 GitHub。

本地预览地址：

- `http://127.0.0.1:8080/agency_partnership_4x3.html`
- `http://127.0.0.1:8080/agency_partnership_4x3_tc.html`

可在项目根目录运行：

```bash
python3 -m http.server 8080 --directory mobile
```

## 3. PDF 生成方式

安装依赖：

```bash
npm install
```

只生成 4:3 PDF：

```bash
node generate_mobile_pdf.js --4x3-only
```

脚本当前使用以下策略保证 PDF 尽量匹配本地预览：

- 使用 `1280 x 720` 浏览器视口渲染，与当前预览环境一致。
- 4:3 页面在该视口中为 `960 x 720`，脚本精确裁取页面区域。
- 最终 PDF 页面为 `1600 x 1200`，保持严格 4:3 比例。
- 使用 `deviceScaleFactor: 2` 进行 Retina 级截图，避免 PDF 放大后粗体模糊或边缘变厚。
- 截图前等待 `document.fonts.ready` 和连续两次浏览器布局刷新。
- PDF 输出时隐藏语言切换按钮。
- 对极限大字页面保留少量 PDF 专用安全修正，避免阅读器缩放后出现遮挡或行距过紧；这些修正不改变 HTML 预览。

脚本依赖本机 Google Chrome，默认路径为：

```text
/Applications/Google Chrome.app/Contents/MacOS/Google Chrome
```

## 4. 核心设计语言

### 4.1 整体风格

- 画布为 4:3，深黑背景配 iFanzy 黄色与白色正文。
- 页面以左右布局为主：左侧为大标题和核心叙事，右侧为结构化模块。
- 视觉要自信、直接、饱满，不使用松散的营销页留白。
- 模块圆角统一以 `10px` 为标准，避免不同页面圆角风格不一致。
- 黄色用于核心标题、关键数据和价值结论；白色用于主要说明；灰色只用于次要信息。

### 4.2 “大填充”设计规范

“大填充”是本项目最重要的模块设计原则：主标题应主动占据模块的上方和横向空间，使页面在 4:3 画布中有力量感，而不是把小文字放在大框里。

- 模块主标题优先靠左、靠上，并尽可能横向填充模块。
- 主标题字号以最长标题能安全容纳为基准，同组模块保持一致，不允许明显忽大忽小。
- 数据类主标题如 `$500B+`、`70M+`、`24/7`、`AI` 应高对比、等视觉重量、横向饱满。
- 主标题和副文案左边对齐；副文案不得漂移，也不能与主标题贴得过紧。
- 模块右侧和上方不应保留大片无意义空白。若持续显空，优先放大主标题，其次调整模块宽度或高度。
- 模块尺寸应适配内容，但同组模块必须保持整齐的行列关系和一致视觉重量。
- 字间距必须自然。禁止通过横向压扁、拉伸字体来伪造填充效果。
- 不能为了填充牺牲完整展示：不得裁切、遮挡、超出边框或产生尴尬分行。
- 英文长标题可做语义分行；被明确要求一行展示的标题必须保持一行。
- 中文大字要同时检查行距和字形上下边界，避免粗体在 PDF 中视觉相碰。

### 4.3 左右视觉平衡

- 左侧大标题与右侧模块应具有接近的视觉重量。
- 右侧模块变窄或缩小时，应把释放的空间用于左侧标题，而不是留下空洞间距。
- 左右间距应紧凑但明确，不能让文字字形进入模块边框。
- 页面视觉重心偏左或偏右时，优先调整整体网格、列宽和间距，不单独用随意位移补救。

### 4.4 文本和模块细节

- 重点文本使用黄色，主要说明使用白色。
- 模块标题与副文案之间使用稳定间距或细分隔线。
- 同组模块标题、正文、边距和分隔线位置要一致。
- 多模块页面应保持统一圆角、边框、背景透明度和阴影强度。
- 文字较多的模块可增加高度、收缩宽度或采用两列信息区，但不能把正文缩到难以阅读。
- 不在卡片中嵌套装饰性卡片，不使用无意义的圆角标签。

## 5. 中英文排版原则

- 中英文版本共享结构和设计语言，但允许根据字宽分别调整列宽、模块宽度和字号。
- 不要求两个语言版本逐像素相同，要求视觉重量、信息层级和模块节奏一致。
- 中文通常可使用更大的字号，但必须为繁体字形保留足够上下空间。
- 英文按语义断行，避免孤立介词、单词尾部被边框挡住或长词进入右侧模块。
- 强制分行使用明确的 `<br>` 或块级 `span`，不要依赖浏览器随机换行。

## 6. PDF 与预览一致性注意事项

即使页面比例相同，外层浏览器视口不同也会触发 `vw`、`vh`、`clamp()`、媒体查询和容器查询，导致字号、间距和模块尺寸变化。因此不能直接用任意 4:3 浏览器视口生成 PDF。

本项目必须继续使用脚本中的“预览视口渲染 + 裁取 4:3 页面”方式。不要恢复为直接使用 `1600 x 1200` 浏览器视口截图。

另外需要注意：

- PDF 当前由高分辨率页面截图组成，视觉一致性优先，文本不一定可选中。
- 字体未加载完成时截图会产生字宽、行高和换行差异。
- PDF 阅读器缩放会放大低分辨率截图的锯齿和粗体边缘，因此必须保留 2x 渲染。
- 极限大字必须留出小幅安全区，浏览器中刚好贴边的效果在 PDF 中风险较高。

## 7. 交付前检查清单

- 中英文均为 14 页。
- PDF 页面尺寸为 `1600 x 1200`，比例为 4:3。
- 页面没有空白、错页或重复页。
- Logo、背景、黄色强调和模块边框完整。
- 所有主标题没有被卡片、边框或页面边缘遮挡。
- 强制一行标题没有意外换行。
- 中文粗体没有上下相碰，英文分行没有过紧或尴尬断词。
- 模块主标题字号在同组内一致，且符合“大填充”效果。
- 模块右侧和底部没有明显无意义空白。
- PDF 重点复检至少包括首页、模块密集页、合作模式页、竞争优势页和最后一页。

推荐检查命令：

```bash
pdfinfo outputs/pdf/agency_partnership_4x3_en.pdf
pdfinfo outputs/pdf/agency_partnership_4x3_tc.pdf
pdftoppm -png outputs/pdf/agency_partnership_4x3_en.pdf tmp/en-page
pdftoppm -png outputs/pdf/agency_partnership_4x3_tc.pdf tmp/tc-page
```

## 8. GitHub 同步原则

- 提交前先执行 `git fetch origin`，确认远端没有新的并行提交。
- 同步时同时更新 HTML、PDF 脚本、最新 PDF 和本交接文档。
- 不要提交临时截图、PDF 渲染检查图或本地缓存。
- 未收到明确指令时，不主动提交或推送。
- 若远端已更新，先检查差异并在保留当前设计成果的前提下整合，不要使用破坏性重置。
